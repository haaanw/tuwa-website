## Summary

Phase 08.1 replaces the bento grid `FeatureGrid.astro` with an iPod-style SVG click wheel. The wheel renders five feature segments as annulus arc `<path>` elements arranged radially at 72° intervals around a central disc. The entire `<g>` group rotates via CSS `transform: rotate()` in response to drag gestures (Pointer Events API) and direct segment clicks. A center HTML overlay (absolutely positioned over the SVG, not part of the rotating group) displays the selected feature's content and deep-dive link.

The implementation is entirely vanilla JS + SVG + CSS inside a single `FeatureGrid.astro` rewrite — no external libraries. This aligns perfectly with the project's "only two islands" philosophy and the `is:inline` pattern already established for the AnimationController and StickyScrollController.

The most complex technical challenges are: (1) correct SVG annulus arc path math for ring segments, (2) angle tracking during drag using `atan2` relative to wheel center, (3) momentum/inertia simulation using velocity sampling on `pointerup`, and (4) transform-origin anchoring for the rotating SVG group. All four have established, well-documented patterns that can be implemented in under 200 lines of vanilla JS.

**Primary recommendation:** Build the wheel as a static SVG (arcs computed at build time from fixed geometry constants, rendered as inline paths in the Astro component) with a single `is:inline` script block handling all runtime interaction. No external libraries needed.

## Standard Stack

### Core

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| Inline SVG | N/A (browser native) | Wheel ring visual, arc segments | Only approach giving precise arc geometry with CSS transform-rotate on a `<g>` group. Canvas would require manual hit-testing. CSS `clip-path` cannot produce interactive arc buttons. |
| Pointer Events API | Browser native | Unified mouse + touch drag | Replaces both `MouseEvent` and `TouchEvent` with one handler set. `setPointerCapture` keeps events flowing if the pointer leaves the SVG boundary during a fast drag. Fully supported in all modern browsers. [VERIFIED: MDN Web Docs] |
| CSS `transform: rotate()` + `transition` | Browser native | Snap animation for click-to-select | Cheapest path — no JS animation loop needed for the final snapped state. Compositor-only, 60fps on mobile. |
| `requestAnimationFrame` | Browser native | Inertia decay animation | Standard pattern for velocity-based momentum in vanilla JS. Already used by AnimationController and counter animation in this codebase. |
| `Math.atan2(dy, dx)` | JS built-in | Pointer angle relative to wheel center | Returns angle in radians from center to pointer. Delta between successive `pointermove` events gives rotation increment. [ASSUMED — well-established trigonometry; no verification needed] |

### Supporting

| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| `window.matchMedia('(prefers-reduced-motion: reduce)')` | Browser native | Suppress momentum + transitions | Check once at script init, store as `reducedMotion` boolean — same pattern as BaseLayout AnimationController |
| CSS custom properties (design tokens) | Already in global.css | All visual values | Never hardcode px/color values — reference tokens by name |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Inline SVG arcs | CSS `clip-path: polygon()` for segments | CSS approach cannot produce an annulus ring with inner cutout; hit-testing on button elements is unreliable with complex clip paths. SVG is the correct tool. |
| Pointer Events API | Split `MouseEvent` + `TouchEvent` | Pointer Events unifies both, handles `setPointerCapture` for out-of-bounds drag, and is the modern standard. No reason to use the split approach. |
| rAF inertia loop | CSS `transition` with cubic-bezier overshoot | CSS transition cannot be interrupted mid-flight by a new drag gesture. rAF loop gives full control over the inertia curve and can be cancelled instantly on `pointerdown`. |
| interact.js (external library) | Vanilla Pointer Events | interact.js is 50KB+ and designed for general-purpose drag/resize. The wheel needs ~60 lines of pointer tracking code. Adding a library for this is unnecessary complexity. [VERIFIED: npm registry size] |

**Installation:** No additional packages required. This phase is zero new dependencies.

### Pattern 1: SVG Annulus Arc Path (Ring Segment)

**What:** Generate the `d` attribute for a ring (annulus) segment spanning a given angular range at given inner and outer radii.

**When to use:** Once for each of the 5 feature segments (computed at build time — hardcoded in Astro component as static SVG paths).

```javascript
// Source: Verified via logrocket.com/interactive-svg-circle-of-fifths (segmentPath formula)
// Adapted for annulus: uses inner + outer radius instead of single radius

function polarToCartesian(cx, cy, r, angleDeg) {
  const rad = (angleDeg - 90) * Math.PI / 180; // -90 so 0° = 12 o'clock
  return {
    x: cx + r * Math.cos(rad),
    y: cy + r * Math.sin(rad),
  };
}

function annulusArcPath(cx, cy, outerR, innerR, startDeg, endDeg) {
  const outerStart = polarToCartesian(cx, cy, outerR, startDeg);
  const outerEnd   = polarToCartesian(cx, cy, outerR, endDeg);
  const innerStart = polarToCartesian(cx, cy, innerR, startDeg);
  const innerEnd   = polarToCartesian(cx, cy, innerR, endDeg);
  const largeArc = endDeg - startDeg > 180 ? 1 : 0;
  return [
    `M ${outerStart.x} ${outerStart.y}`,
    `A ${outerR} ${outerR} 0 ${largeArc} 1 ${outerEnd.x} ${outerEnd.y}`,
    `L ${innerEnd.x} ${innerEnd.y}`,
    `A ${innerR} ${innerR} 0 ${largeArc} 0 ${innerStart.x} ${innerStart.y}`,
    'Z',
  ].join(' ');
}
```

**Geometry constants for desktop viewBox (480×480, center at 240,240):**
- `outerR = 210`, `innerR = 170` → 40px radial depth (label zone per UI-SPEC)
- Segment 0 (Recovery): startDeg = -36, endDeg = 36 (centered at 0° = 12 o'clock)
- Each subsequent segment adds 72°
- Use a 2° gap between segments (startDeg + 1, endDeg - 1) for visual separation

**Icon + label group center point** (midpoint of each segment's midline radius):
```javascript
// midRadius = (outerR + innerR) / 2 = 190
// midAngle = startDeg + (endDeg - startDeg) / 2
const iconPos = polarToCartesian(240, 240, 190, midAngle);
```

Since these are static coordinates, compute them once and hardcode as SVG `transform="translate(x, y)"` attributes in the Astro template.

### Pattern 2: Pointer Events Drag Rotation

**What:** Track cumulative rotation angle during a drag gesture using the angle between the wheel center and the pointer position.

**When to use:** In the `is:inline` script inside `FeatureGrid.astro`.

```javascript
// Source: [ASSUMED — standard atan2 polar tracking pattern, widely documented]
// setPointerCapture: [VERIFIED: MDN Web Docs - Element.setPointerCapture()]

const svgEl = document.getElementById('feature-wheel-svg');
const groupEl = document.getElementById('wheel-segments-group');
const cx = 240, cy = 240; // viewBox center

let isDragging = false;
let startAngle = 0;
let currentRotation = 0; // total rotation applied to <g>
let lastAngle = 0;
let velocitySamples = []; // [{angle, time}] — last 3 pointermove events

function getAngleFromCenter(event) {
  const rect = svgEl.getBoundingClientRect();
  // Map client coords to SVG viewBox coords
  const scaleX = 480 / rect.width;
  const scaleY = 480 / rect.height;
  const dx = (event.clientX - rect.left) * scaleX - cx;
  const dy = (event.clientY - rect.top)  * scaleY - cy;
  return Math.atan2(dy, dx) * (180 / Math.PI);
}

svgEl.addEventListener('pointerdown', (e) => {
  isDragging = true;
  svgEl.setPointerCapture(e.pointerId); // keep events even if pointer leaves SVG
  startAngle = getAngleFromCenter(e) - currentRotation;
  velocitySamples = [];
  // Remove snap transition during drag
  groupEl.style.transition = 'none';
});

svgEl.addEventListener('pointermove', (e) => {
  if (!isDragging) return;
  const angle = getAngleFromCenter(e);
  const newRotation = angle - startAngle;
  const delta = newRotation - currentRotation;
  currentRotation = newRotation;
  groupEl.style.transform = `rotate(${currentRotation}deg)`;
  // Sample velocity
  velocitySamples.push({ angle: currentRotation, time: performance.now() });
  if (velocitySamples.length > 3) velocitySamples.shift();
});

svgEl.addEventListener('pointerup', () => {
  if (!isDragging) return;
  isDragging = false;
  applyInertiaAndSnap();
});
```

**Key insight — SVG coordinate scaling:** The pointer's `clientX/Y` are in CSS pixels. The SVG's viewBox uses 480 units. Must scale using `rect.width / 480` to correctly map pointer position to the wheel center in viewBox coordinates. Missing this scaling step causes angle drift at different viewport widths.

### Pattern 3: Momentum/Inertia + Snap

**What:** On drag release, apply velocity-based overshoot then snap to nearest 72° multiple.

```javascript
// Source: [ASSUMED — standard velocity sampling pattern for inertia simulation]
const reducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
const SEGMENT_ANGLE = 72; // 360 / 5
const MAX_OVERSHOOT = 30; // degrees, per UI-SPEC D-06

function applyInertiaAndSnap() {
  if (reducedMotion || velocitySamples.length < 2) {
    snapToNearest();
    return;
  }
  // Calculate velocity (degrees per ms)
  const first = velocitySamples[0];
  const last  = velocitySamples[velocitySamples.length - 1];
  const dt = last.time - first.time;
  const velocity = dt > 0 ? (last.angle - first.angle) / dt : 0; // deg/ms

  let inertiaAngle = currentRotation;
  let v = velocity;
  const DECAY = 0.92; // multiply velocity each frame (~60fps)
  const FRAME_MS = 16.67;

  function inertiaFrame() {
    v *= DECAY;
    inertiaAngle += v * FRAME_MS;
    // Clamp overshoot relative to nearest snap point
    const nearest = Math.round(inertiaAngle / SEGMENT_ANGLE) * SEGMENT_ANGLE;
    if (Math.abs(inertiaAngle - nearest) > MAX_OVERSHOOT) {
      inertiaAngle = nearest + Math.sign(inertiaAngle - nearest) * MAX_OVERSHOOT;
      v = 0; // stop inertia
    }
    currentRotation = inertiaAngle;
    groupEl.style.transform = `rotate(${currentRotation}deg)`;
    if (Math.abs(v) > 0.1) {
      requestAnimationFrame(inertiaFrame);
    } else {
      snapToNearest();
    }
  }
  requestAnimationFrame(inertiaFrame);
}

function snapToNearest() {
  const nearest = Math.round(currentRotation / SEGMENT_ANGLE) * SEGMENT_ANGLE;
  currentRotation = nearest;
  const activeIndex = (((-nearest / SEGMENT_ANGLE) % 5) + 5) % 5;
  // Apply CSS transition for the snap
  groupEl.style.transition = reducedMotion ? 'none' : 'transform 200ms ease-out';
  groupEl.style.transform = `rotate(${nearest}deg)`;
  updateActiveSegment(activeIndex);
}
```

**Sign convention:** Rotating the wheel clockwise (positive CSS rotation) moves features counter-clockwise visually. The active index maps as: `activeIndex = ((-rotation / 72) % 5 + 5) % 5`. The `+5) % 5` normalization handles negative modulo in JS.

### Pattern 4: SVG Group Rotation — transform-origin

**What:** Rotating the SVG `<g>` group about the wheel center (not the SVG origin).

**Critical gotcha:** CSS `transform-origin` on SVG elements uses the viewBox coordinate system by default (`transform-box: view-box`). The default origin is `0 0` (top-left corner), NOT the group's center.

```css
/* Source: [VERIFIED: MDN transform-box, CSS-Tricks SVG transforms article] */

#wheel-segments-group {
  /* transform-box: fill-box uses element bounding box */
  /* transform-box: view-box uses SVG viewBox (default) */

  /* Option A (recommended for this case): explicit origin in viewBox coords */
  transform-origin: 240px 240px; /* wheel center in viewBox units */
  transform-box: view-box;

  /* Option B: SVG attribute transform="rotate(deg, cx, cy)" */
  /* <g transform="rotate(0, 240, 240)"> — set via JS: */
  /* groupEl.setAttribute('transform', `rotate(${angle}, 240, 240)`) */
}
```

**Recommendation:** Use the SVG `transform` attribute approach (`setAttribute('transform', 'rotate(N, 240, 240)')`) rather than CSS `transform`. This avoids `transform-box` browser inconsistencies entirely and is the most reliable cross-browser approach for SVG group rotation. The CSS `transition` for the snap animation must then be applied via the Web Animations API (`groupEl.animate([...], { duration: 200, easing: 'ease-out' })`), or use a CSS approach only for the final snap step.

**Simpler hybrid:** Apply rotation as SVG attribute during drag (via `setAttribute`), switch to CSS class with `transition` for the snap animation only.

### Pattern 5: Click-to-Select (Segment Buttons)

**What:** Each segment has a `<button>` (overlaid on the SVG arc) or a `<foreignObject>` element for accessibility. The simpler and more compatible approach is `<button>` elements positioned via CSS `position: absolute` over the wheel.

**Architecture decision:** The SVG arc paths handle the visual ring. Separate `<button>` elements with `position: absolute` are layered over each segment's midpoint area. This avoids `<foreignObject>` rendering inconsistencies across browsers.

```html
<!-- Wheel container: position:relative wrapper -->
<div class="wheel-container" style="position: relative; width: 480px; height: 480px;">
  <svg id="feature-wheel-svg" viewBox="0 0 480 480" ...>
    <!-- Arc paths, icons (aria-hidden="true"), labels rendered here -->
  </svg>
  <!-- Segment buttons absolutely positioned over arc midpoints -->
  <!-- Their clip-path matches the arc shape for visual accuracy -->
  <!-- Positioned at icon+label group center coords -->
  <button class="wheel-segment-btn" data-index="0" aria-pressed="true"
    style="position: absolute; left: 195px; top: 20px; width: 90px; height: 44px;">
    <!-- visually hidden label for screen readers -->
    <span class="sr-only">Recovery Scoring — HRV and readiness scoring</span>
  </button>
  <!-- ... 4 more buttons ... -->

  <!-- Center overlay: NOT inside SVG, does NOT rotate -->
  <div class="wheel-center-overlay" ...>
    <!-- icon, title, description, Explore link -->
  </div>
</div>
```

**Alternative considered:** Render buttons as SVG `<foreignObject>`. Rejected — `<foreignObject>` has known rendering bugs in some Safari versions and complicates the rotation group (buttons must not rotate with the arc group).

### Pattern 6: Center Overlay Content Swap

**What:** On segment activation, swap the center preview content. Crossfade via opacity.

```javascript
// Source: [ASSUMED — standard crossfade pattern]
const features = [
  { title: 'Recovery Scoring', desc: '...', href: '/features/recovery-scoring', icon: '...' },
  // ... 4 more
];

function updateActiveSegment(index) {
  // Update aria-pressed on buttons
  document.querySelectorAll('.wheel-segment-btn').forEach((btn, i) => {
    btn.setAttribute('aria-pressed', i === index ? 'true' : 'false');
  });
  // Update active visual class on SVG paths
  document.querySelectorAll('.wheel-arc').forEach((arc, i) => {
    arc.classList.toggle('is-active', i === index);
  });

  // Center content swap
  const centerEl = document.querySelector('.wheel-center-content');
  if (reducedMotion) {
    updateCenterContent(centerEl, features[index]);
  } else {
    centerEl.style.opacity = '0';
    centerEl.style.transition = 'opacity 75ms ease-out';
    setTimeout(() => {
      updateCenterContent(centerEl, features[index]);
      centerEl.style.opacity = '1';
    }, 75);
  }
}
```

### Anti-Patterns to Avoid

- **Rotating the center overlay with the SVG `<g>`:** The center preview must stay fixed. Keep it as a sibling HTML element, not inside the SVG rotating group.
- **Using `transform: rotate()` CSS on SVG `<g>` without setting `transform-origin`:** The group will rotate about the SVG's `0,0` origin (top-left), producing incorrect behaviour. Always specify the center explicitly.
- **Reading `clientX/Y` directly without scaling to viewBox coordinates:** At non-native SVG sizes (all responsive breakpoints), raw client coords produce wrong angle calculations. Always scale by `viewBox_dimension / rect.dimension`.
- **Using `touchstart/touchmove/touchend` separately from mouse events:** Pointer Events API handles both. Using both event systems creates duplicate handler registration and conflicts.
- **Hardcoding pixel values for button position:** Compute button positions from the same `polarToCartesian` function used for SVG arcs. At different breakpoints, the SVG scales via CSS `width` but the viewBox stays `0 0 480 480` — all SVG geometry is self-consistent.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Arc path math for each segment | Custom geometry code per feature | Single `annulusArcPath()` function + loop | Arc path parameters are computed identically for all 5 segments; parameterize once |
| Inertia physics | Complex spring simulation | Simple velocity decay loop (DECAY constant × velocity per frame) | The UI-SPEC caps overshoot at 30° and inertia at 300ms — this is achievable with a 10-line rAF loop, not a physics engine |
| Pointer unification across devices | Separate mouse + touch handlers | Pointer Events API (`pointerdown/move/up`) | Browser-unified, already normalises touch/mouse/stylus |
| Focus management | Manual tabIndex manipulation | Existing `:focus-visible` + semantic `<button>` elements | Native buttons receive focus automatically; existing `--color-accent` focus ring in global.css applies without extra code |

**Key insight:** The iPod click wheel interaction looks complex but decomposes into four independent concerns (arc geometry, angle tracking, inertia decay, content swap) each requiring ~15–30 lines of vanilla JS.

## Common Pitfalls

### Pitfall 1: SVG viewBox Coordinate Mismatch During Pointer Tracking

**What goes wrong:** At breakpoints where the SVG renders at less than 480px wide, `event.clientX` reports CSS pixel coordinates but the SVG geometry is defined in 480-unit viewBox space. Using raw client coords for `atan2` produces angle drift proportional to the CSS scale factor.

**Why it happens:** SVG's `viewBox` creates an internal coordinate system that scales independently of the rendered CSS size.

**How to avoid:** Always compute the scale factor: `scaleX = 480 / svgEl.getBoundingClientRect().width`. Apply before subtracting the wheel center: `dx = (event.clientX - rect.left) * scaleX - 240`.

**Warning signs:** Rotation feels sluggish or "sticky" on mobile vs desktop. The wheel snaps to wrong positions when the viewport is narrow.

### Pitfall 2: transform-origin for SVG Group Rotation

**What goes wrong:** `groupEl.style.transform = 'rotate(45deg)'` on a `<g>` element rotates about the SVG `0,0` origin (top-left corner of viewBox), not the wheel center. The ring visually flies off-screen.

**Why it happens:** CSS `transform-box` defaults to `view-box`, meaning transform-origin is relative to the SVG viewport, not the element's own bounding box.

**How to avoid:** Use the SVG presentation attribute for rotation: `groupEl.setAttribute('transform', `rotate(${angle}, 240, 240)`)`. This syntax includes the center of rotation explicitly and is cross-browser safe. For the CSS transition snap, use `el.animate([{ transform: `rotate(${from}deg)` }, { transform: `rotate(${to}deg)` }], ...)` with explicit transform-origin set in CSS.

**Warning signs:** Wheel visually disappears or jumps to a corner on first rotation.

### Pitfall 3: Buttons Overlap and Block Drag Gestures

**What goes wrong:** `<button>` elements positioned over arc midpoints capture `pointerdown` before the drag handler on the SVG element, breaking drag-to-rotate.

**Why it happens:** Event propagation — the button is on top in the DOM stacking order.

**How to avoid:** Attach `pointerdown` to the button as well (or its parent container). In the `pointerdown` handler, if the intent becomes a drag (cumulative movement > 5px threshold), call `e.preventDefault()` on subsequent moves and cancel the click. Use the "drag threshold" pattern: only emit a click if the total drag distance is under 5px.

**Warning signs:** Dragging starts from a segment button position and the wheel doesn't rotate.

### Pitfall 4: Angle Discontinuity at ±180°

**What goes wrong:** `atan2` returns values in the range [-180°, 180°]. When the pointer crosses the ±180° boundary (right side of wheel at 3 o'clock), the delta jumps from +179° to -180°, causing the wheel to suddenly spin backward.

**Why it happens:** `atan2` wraps at ±π radians.

**How to avoid:** Normalize the angle delta. After computing `delta = newAngle - prevAngle`, apply:
```javascript
if (delta > 180) delta -= 360;
if (delta < -180) delta += 360;
```
This collapses any large jump into the equivalent short-arc rotation.

**Warning signs:** Wheel spins 359° in the wrong direction when dragging across the 3 o'clock position.

### Pitfall 5: Removing the bento grid CSS breaks feature deep-dive pages

**What goes wrong:** `.feature-card` base class is referenced in `global.css` and used across multiple pages. The UI-SPEC notes that `.bento-grid` and `.bento-hero-card` should be removed, but `.feature-card` may be used in feature deep-dive pages.

**Why it happens:** CSS classes are global; removal affects all pages.

**How to avoid:** Before removing `.feature-card`, grep for all usages across the codebase:
```bash
grep -r "feature-card" src/
```
Only remove `.bento-grid` and `.bento-hero-card`. Keep `.feature-card` unless confirmed unused.

**Warning signs:** Feature deep-dive pages lose card styling after the phase ships.

### Pitfall 6: prefers-reduced-motion guard scope

**What goes wrong:** Applying `prefers-reduced-motion` only to the entrance animation but forgetting it for the inertia loop and center crossfade. Users with motion sensitivity still experience inertia spin.

**Why it happens:** Multiple animation paths need individual guards.

**How to avoid:** Set `const reducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches` once at script init. Reference this boolean in all three places: (1) `pointerup` inertia trigger, (2) center content crossfade, (3) click-to-select CSS transition.

### Annulus Arc Path for 5 Segments (static geometry at build time)

```javascript
// Source: Adapted from logrocket.com/interactive-svg-circle-of-fifths segmentPath formula
// Geometry: viewBox 480x480, center (240,240), outerR=210, innerR=170, gap=2deg

const CX = 240, CY = 240, OUTER_R = 210, INNER_R = 170;
const GAP_DEG = 2; // visual gap between segments
const SEGMENT_DEG = 72;

function polarToCartesian(cx, cy, r, angleDeg) {
  const rad = (angleDeg - 90) * Math.PI / 180; // 0° = 12 o'clock
  return { x: cx + r * Math.cos(rad), y: cy + r * Math.sin(rad) };
}

function arcPath(startDeg, endDeg) {
  const s0 = polarToCartesian(CX, CY, OUTER_R, startDeg + GAP_DEG/2);
  const e0 = polarToCartesian(CX, CY, OUTER_R, endDeg - GAP_DEG/2);
  const s1 = polarToCartesian(CX, CY, INNER_R, startDeg + GAP_DEG/2);
  const e1 = polarToCartesian(CX, CY, INNER_R, endDeg - GAP_DEG/2);
  return [
    `M ${s0.x.toFixed(3)} ${s0.y.toFixed(3)}`,
    `A ${OUTER_R} ${OUTER_R} 0 0 1 ${e0.x.toFixed(3)} ${e0.y.toFixed(3)}`,
    `L ${e1.x.toFixed(3)} ${e1.y.toFixed(3)}`,
```

## Open Questions (RESOLVED)

1. **Button positioning approach at mobile breakpoints**
   - What we know: The SVG scales via CSS `width: 280px` (mobile) vs `480px` (desktop). viewBox stays `0 0 480 480`.
   - What's unclear: Whether `<button>` elements positioned as percentage of container will correctly overlap arc midpoints at all three breakpoint sizes.
   - Recommendation: During Wave 1 implementation, position buttons using percentage-based `left`/`top` relative to the wheel container div (compute as `buttonX / 480 * 100%`). If misalignment occurs, add a `ResizeObserver` to recompute button positions.

2. **`.feature-card` CSS class usage in feature pages**
   - What we know: UI-SPEC says remove `.bento-grid` and `.bento-hero-card`. `.feature-card` "may be retained."
   - What's unclear: Whether any feature deep-dive page uses `.feature-card` outside of FeatureGrid.
   - Recommendation: Wave 0 task should grep for `feature-card` across `src/` before removing anything.

3. **SVG attribute vs CSS transform for rotation during animation**
   - What we know: CSS `transform` on SVG `<g>` has `transform-origin` cross-browser complications. SVG `transform` attribute is reliable.
   - What's unclear: Whether `el.animate()` WAAPI works correctly with SVG `transform` attribute (vs CSS property).
   - Recommendation: Use CSS approach (`groupEl.style.transform = 'rotate(Ndeg)'` with explicit `transform-origin: 240px 240px; transform-box: view-box` in CSS) for the snap animation, and SVG attribute for drag. Test during Wave 1 and fall back to full attribute approach if CSS snap animation misbehaves.
