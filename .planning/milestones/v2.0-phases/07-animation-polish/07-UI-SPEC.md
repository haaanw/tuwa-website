## Animation Contract

This is the primary contract for Phase 7. All values are derived from CONTEXT.md decisions D-01 through D-09.

### Existing Animation (unchanged, source: `src/styles/global.css`)

| Property | Value |
|----------|-------|
| Keyframe | `fade-up`: `opacity: 0 → 1`, `translateY(16px → 0)` |
| Duration | 400ms |
| Easing | `ease-out` |
| Trigger | `.js-enabled [data-animate].is-visible` (IntersectionObserver in BaseLayout) |
| No-JS fallback | All content visible at full opacity (no transform applied without JS) |
| Reduced-motion | `@media (prefers-reduced-motion: reduce)` block intentionally empty — elements render static |

### New: Stagger Delays on FeatureGrid (ANIM-03)

**Decision source: CONTEXT.md D-01, D-02**

- Move `data-animate` from `<section>` to each `<li>` element in FeatureGrid.
- Add `data-animate-delay` attribute to each `<li>` with explicit millisecond values.
- Stagger interval: **100ms** per card (within the 80-120ms range from D-01).
- Total stagger window: 5 cards × 100ms = **400ms** total cascade.

| Card | data-animate-delay value |
|------|--------------------------|
| Card 1 (Recovery Scoring) | 0ms |
| Card 2 (Workload Tracking) | 100ms |
| Card 3 (Smart Templates) | 200ms |
| Card 4 (Cold-Start) | 300ms |
| Card 5 (Coach + Athlete) | 400ms |

CSS rule to add in `global.css`:
```css
@media (prefers-reduced-motion: no-preference) {
  .js-enabled [data-animate][data-animate-delay] {
    animation-delay: attr(data-animate-delay ms, 0ms);
  }
}
```
Note: If `attr()` with type hints lacks browser support, AnimationController reads `data-animate-delay` and applies `element.style.animationDelay` in JS before adding `.is-visible`.

Stagger applies to **FeatureGrid only** (D-02). All other `data-animate` elements keep simultaneous fade-up behavior.

### New: Hero Entrance Choreography (ANIM-04)

**Decision source: CONTEXT.md D-03, D-04, D-08, D-09**

- Animates on page load. No IntersectionObserver. Uses CSS `animation` property directly.
- Sequence: headline → subtitle → device mockup. Text-first, device as payoff.
- All three use the existing `fade-up` keyframe (opacity + translateY), no new keyframes needed.
- Reduced-motion: add `@media (prefers-reduced-motion: no-preference)` guard around hero animations. At `reduce`, all hero elements render at full opacity with no transform.

| Element | CSS class to add | Animation delay | Duration | Easing |
|---------|-----------------|-----------------|----------|--------|
| `<h1>` headline | `hero-headline` | 0ms | 400ms | ease-out |
| `<p>` subtitle | `hero-subtitle` | 150ms | 400ms | ease-out |
| Device mockup wrapper `<div>` | `hero-device` | 350ms | 500ms | ease-out |

Device mockup additionally scales from `scale(0.96)` to `scale(1)` to give the "assembles" feel (D-03). Add a `hero-device-enter` keyframe:

```css
@keyframes hero-device-enter {
  from {
    opacity: 0;
    transform: perspective(1600px) rotateY(-2deg) rotateX(1deg) translateY(20px) scale(0.96);
  }
  to {
    opacity: 1;
    transform: perspective(1600px) rotateY(-2deg) rotateX(1deg) translateY(0) scale(1);
  }
}
```

The inline `transform: perspective(...)` on the wrapper `<div>` in Hero.astro must be moved into the keyframe `to` state to avoid fighting the animation `from` state.

Hero animation classes are applied unconditionally in HTML. The `@media (prefers-reduced-motion: no-preference)` wrapper ensures they only animate under motion-safe conditions.

### New: Sticky Scroll Showcase on Recovery Scoring (ANIM-05)

**Decision source: CONTEXT.md D-05, D-06, D-07**

**Layout: Sticky device + scrolling text steps (Oura Ring pattern)**

- Desktop: two-column layout. Left column: sticky device frame pinned with `position: sticky; top: 24px`. Right column: 3 scroll step blocks stacked vertically, each tall enough to trigger step transitions.
- Mobile: collapses to standard stacked layout — device frame at top, step content blocks below in sequence. No sticky behavior on mobile (below `md` breakpoint).

**3 scroll steps (grouping of existing recovery-scoring.astro subsections):**

| Step | Label | Content | Screenshot |
|------|-------|---------|------------|
| Step 1 | "Your daily readiness score" | Recovery zones overview + what the score means | `recovery.png` (primary, existing asset) |
| Step 2 | "Built from six dimensions" | Data sources: HRV, sleep, heart rate, training context, resting HR, HRV trend | Same `recovery.png` or variant showing data breakdown — use same if no second screenshot exists |
| Step 3 | "Calibrated to you" | Personal baseline section — how Tuwa adapts over time | Same `recovery.png` — use content text as the differentiator |

Note: Claude is given discretion to refine step content grouping during implementation (D-07).

**Screenshot swap mechanism:**

- Crossfade between step screenshots using CSS `opacity` transitions (300ms ease-out).
- Implement via IntersectionObserver watching each `.scroll-step` element.
- When a step enters the viewport center (threshold: 0.5), it becomes `is-active`, and the device frame swaps to that step's screenshot with a 300ms crossfade.
- No JavaScript scroll events. IntersectionObserver only.
- If all steps use `recovery.png`, the swap mechanism still runs but swaps to the same image (no visible flicker — this is acceptable).

**Sticky scroll step indicator:**

- 3 dots in the device frame or adjacent to it. Inactive: `--color-text-3`. Active: `--color-accent`.
- Dots are decorative only — not interactive, not keyboard-navigable.

**No CLS rule (ANIM-03 success criterion 4):**

- Sticky scroll layout uses `transform` and `opacity` only. No animation touches `height`, `margin`, `top`, or `left` with layout recalculation.
- Device frame dimensions are fixed (CSS `aspect-ratio` from DeviceFrame.astro). Screenshot swap via `opacity` crossfade does not cause layout shift.

## Accessibility Contract

All animation additions must satisfy these constraints:

1. `prefers-reduced-motion: reduce` — zero motion. All new animations wrapped in `@media (prefers-reduced-motion: no-preference)`. Hero elements render at full opacity immediately at `reduce`.
2. No-JS fallback — hero headline, subtitle, and device are visible at full opacity without JS. Add `hero-headline`, `hero-subtitle`, `hero-device` CSS only inside the `@media (prefers-reduced-motion: no-preference)` block, so non-JS users see content instantly.
3. Stagger elements are content, not decorative — `<li>` items in FeatureGrid must remain visible even if observer fires late.
4. Sticky scroll step indicator dots are `aria-hidden="true"` — purely decorative.
5. Sticky scroll on mobile collapses gracefully — all content readable without the sticky/scroll interaction.
